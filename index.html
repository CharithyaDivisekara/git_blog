What is Git?

Git is a distributed version control system that helps developers track changes in their code during software development. It was created by Linus Torvalds in 2005 for Linux kernel development.

Think of Git as a time machine for your code - it allows you to go back to previous versions, see who made changes, and collaborate with others without overwriting each other's work.

Why we say Git is a version controller

Git manages and tracks the history of changes to files. Whenever you save a new version of your project, Git takes a snapshot of what all your files look like at that moment and stores a reference to that snapshot.

Importance of Git

  • Enables collaboration among developers
  • Tracks changes and history of projects
  • Allows reverting to previous versions
  • Facilitates branching and experimentation

Key Features of Git

  • Distributed development
  • Branching and merging
  • Staging area
  • Fast and efficient performance

Git Status and Basic Workflow

Checking Git Status

The git status command shows the state of your working directory and staging area:

git status
On branch main
No commits yet
Untracked files:
  (use "git add <file>..." to include in what will be committed)
      index.html
      styles.css

nothing added to commit but untracked files present (use "git add" to track)

The Three Main Areas in Git

1. Working Directory

This is where you modify your files. It represents the current state of your project on your local machine. Files in the working directory can be in one of two states: tracked or untracked.

2. Staging Area (Index)

The staging area is where you prepare changes for the next commit. When you use git add, you're moving changes from the working directory to the staging area. This allows you to carefully curate which changes will be part of the next snapshot.

3. Repository (.git directory)

This is where Git permanently stores the changes as commits. Once you commit changes, they are safely stored in the repository's database. The repository contains all of the commits, branches, tags, and other history of your project.

git status

  (use "git add <file>..." to include in what will be committed)
       newpage.html

Checking Out Specific Commits

You can check out a specific commit using its hash (first 7 characters are usually enough):

git checkout a1b2c3d
Note: switching to 'a1b2c3d'.

You are in 'detached HEAD' state. You can look around, make experimental
changes and commit them, and you can discard any commits you make in this
state without impacting any branches by switching back to a branch.

If you want to create a new branch to retain commits you create, you may
do so (now or later) by using -c with the switch command. Example:

git switch -c <new-branch-name>

Or undo this operation with:

git switch -

Turn off this advice by setting config variable advice.detachedHead to false

HEAD is now at a1b2c3d Add descriptive commit message here

Note: In this detached HEAD state, you'll only see the files as they were at that specific commit. Other files may appear to be deleted, but they're still in your repository's history.

Returning to the Latest Commit

To return to the latest commit in your branch:

git switch -
Switched to branch 'main'

Delete and Rename Operations in Git

Removing Files in Git

There are two ways to remove files in Git, each with different implications:

1. Using rm (Standard Unix Command)

This removes the file from your working directory but doesn't stage the deletion in Git:

$ rm file.txt
# File is removed from working directory
$ git status
On branch main
Changes not staged for commit:
  (use "git add/rm <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
       deleted:   file.txt

After using rm, you need to explicitly stage the deletion with git add or git rm.

2. Using git rm (Git Command)

This removes the file from your working directory and stages the deletion in one step:

$ git rm file.txt
rm 'file.txt'
$ git status
On branch main
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
       deleted:   file.txt

Key Difference: rm only removes the file from your working directory, while git rm removes the file and stages the deletion in one command.

Renaming Files in Git

Similarly, there are two approaches to renaming files:

1. Using mv (Standard Unix Command)

This renames the file in your working directory but doesn't stage the rename in Git:

$ mv oldname.txt newname.txt
# File is renamed in working directory
$ git status
On branch main
Changes not staged for commit:
  (use "git add/rm <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
       deleted:   oldname.txt

Untracked files:
  (use "git add <file>..." to include in what will be committed)
       newname.txt

Git sees this as deleting the old file and creating a new one. You need to stage both changes manually.

2. Using git mv (Git Command)

This renames the file and stages the rename in one step, preserving file history:

$ git mv oldname.txt newname.txt
# File is renamed and change is staged
$ git status
On branch main
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
       renamed:   oldname.txt -> newname.txt

Key Difference: mv only renames the file in your working directory, while git mv renames the file and stages the change, allowing Git to track the file's history across the rename.

Committing the Changes

After using either method, you need to commit the changes:

$ git commit -m "Remove file.txt and rename oldname.txt to newname.txt"
[main 1a2b3c4] Remove file.txt and rename oldname.txt to newname.txt
2 files changed, 0 insertions(+), 0 deletions(-)
delete mode 100644 file.txt
rename oldname.txt => newname.txt (100%)

Note: Git doesn't actually store rename operations. Instead, it detects renames by comparing file contents between commits. Using git mv is essentially a convenience command that performs mv followed by git add for both the old and new filenames.

Undo Operations in Git

Git provides several ways to undo changes at different stages of the workflow. Here are the main undo operations:

1

Discard changes in working directory

Revert a file back to how it was in the last commit:

$ git restore filename.txt
# Changes in working directory are discarded

This command will undo any changes you've made to the file since the last commit, restoring it to the state in the HEAD commit.

2

Unstage changes

Remove a file from the staging area but keep changes in working directory:

$ git restore --staged filename.txt
# File is unstaged but changes are kept
$ git status
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
       modified:   filename.txt

no changes added to commit (use "git add" and/or "git commit -a")

This is useful when you've added a file to the staging area but realize you're not ready to commit it yet.

3

Undo commit but keep changes staged

Move HEAD pointer to previous commit but keep changes in staging area:

$ git reset --soft HEAD~1
# Last commit is undone, changes are staged
$ git status
On branch main
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
       modified:   filename.txt

This command is useful when you want to redo a commit with a different message or add more changes to it.

4

Undo commit and unstage changes

Move HEAD pointer to previous commit and unstage changes:

$ git reset HEAD~1
# Default is --mixed: undo commit and unstage changes
$ git status
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
       modified:   filename.txt

no changes added to commit (use "git add" and/or "git commit -a")

This is the default reset mode. It undoes the commit and keeps the changes in your working directory but not staged.

5

Completely remove commit and discard changes

Completely remove the last commit and all its changes:

$ git reset --hard HEAD~1
# Last commit and all changes are completely removed
$ git status
On branch main
nothing to commit, working tree clean

Warning: This command will permanently delete your changes. Use with caution!

This command should be used carefully as it permanently discards all changes from the undone commit.

Note: The HEAD~1 syntax refers to the commit before the current HEAD (last commit). You can change the number to go back multiple commits (e.g., HEAD~3 to go back 3 commits).

Working with Branches in Git

What are Branches?

Branches in Git are pointers to specific commits. They allow you to work on different versions of your code simultaneously without affecting the main codebase.

Why Use Branches?

Branches are essential for:

  • Parallel development: Work on multiple features simultaneously
  • Isolation: Experiment with new ideas without breaking the main code
  • Collaboration: Multiple developers can work on different features
  • Release management: Maintain different versions of your software
  • Bug fixing: Fix issues without disrupting ongoing development

Basic Branch Commands

Create a new branch

Create a new branch pointing to the current commit:

$ git branch feature-login
# Creates a new branch called 'feature-login'

Switch to a branch

Move to an existing branch:

$ git switch feature-login
Switched to branch 'feature-login'

Create and switch in one command

Create a new branch and immediately switch to it:

$ git switch -c feature-signup
Switched to a new branch 'feature-signup'

The -c flag stands for "create". This is a convenient shortcut for creating and switching to a new branch.

List all branches

Show all branches in your repository:

$ git branch
feature-login
main
* feature-signup
# The asterisk (*) indicates the current branch

Merge a branch

Combine changes from one branch into another:

$ git switch main
Switched to branch 'main'
$ git merge feature-login
Updating a1b2c3d..4d5e6f7
Fast-forward
login.html | 15 +++++++++++++++
1 file changed, 15 insertions(+)
create mode 100644 login.html

Delete a branch

Remove a branch that is no longer needed:

$ git branch -d feature-login
Deleted branch feature-login (was 4d5e6f7).

Branching Strategy: A common approach is to keep the main branch stable and create feature branches for new developments. Once a feature is complete and tested, it can be merged back into the main branch.

Branch Merging in Git

What is Branch Merging?

Merging is the process of combining changes from one branch into another. This allows you to integrate work done in feature branches back into your main development branch.

Basic Merge Command

To merge a branch into your current branch:

$ git switch main
Switched to branch 'main'
$ git merge feature-branch
Updating a1b2c3d..4d5e6f7
Fast-forward
index.html | 5 +++++
1 file changed, 5 insertions(+)

Understanding Three-Way Merge

Git uses a three-way merge algorithm when the branches have diverged (both have new commits since they split):

  • Base: The common ancestor commit of both branches
  • Current Change (Ours): The state of your current branch
  • Incoming Change (Theirs): The state of the branch you're merging in

Git compares these three states to determine how to combine the changes. If changes don't conflict, Git automatically merges them.

Resolving Merge Conflicts

When the same part of a file was modified differently in both branches, Git can't automatically merge and creates a conflict:

$ git merge feature-branch
Auto-merging index.html
CONFLICT (content): Merge conflict in index.html
Automatic merge failed; fix conflicts and then commit the result.

The conflicted file will contain markers showing both versions:

<<<<<<< HEAD
Current branch changes
=======
Incoming branch changes
>>>>>>> feature-branch

Resolving Conflicts in VS Code

VS Code provides a visual interface for resolving merge conflicts with these options:

Accept Current Change

Keep your version (from the current branch) and discard the incoming changes

Accept Incoming Change

Use the incoming version (from the branch being merged) and discard your changes

Accept Both Changes

Keep both versions, one after the other (may require manual editing)

Compare Changes

View a side-by-side comparison to help decide which changes to keep

Manual Resolution

Edit the file manually to create a custom combination of both changes

Completing the Merge

After resolving all conflicts, you need to complete the merge:

$ git add .
# Add all resolved files to staging
$ git commit
[main 7a8b9c0] Merge branch 'feature-branch'

Tip: Git will open your default text editor with a pre-filled commit message for merge commits. You can keep this message or modify it as needed.

Team Collaboration with Git Branches

Real-World Example: Team Project

Let's walk through a practical example of how a team uses Git branches to work on different features simultaneously.

Main Branch
Member A (Login Fix)
Member B (Shipping Fix)
1

Initial Project Setup

$ mkdir ecommerce-project
$ cd ecommerce-project
$ git init
Initialized empty Git repository
$ touch index.html style.css script.js
$ git add .
$ git commit -m "Initial project setup"
[main (root-commit) a1b2c3d] Initial project setup
2

Manager Assigns Tasks

Manager identifies two issues:

  • Member A: Fix login page error
  • Member B: Fix shipping cart calculation error

Both members create separate branches from main:

$ git branch fix-login
$ git branch fix-shipping
3

Parallel Development

Both members work simultaneously on their branches:

Member A (fix-login)
$ git switch fix-login
# Edits login functionality in script.js
$ git commit -am "Fix login validation"
Member B (fix-shipping)
$ git switch fix-shipping
# Edits shipping calculation in script.js
$ git commit -am "Fix shipping calculation"
4

Member A Merges First

Member A completes their fix and merges to main:

$ git switch main
$ git merge fix-login
Updating a1b2c3d..4d5e6f7
Fast-forward
script.js | 3 +++
1 file changed, 3 insertions(+)

Manager reviews and approves the changes

5

Member B Updates Their Branch

Member B pulls the latest changes from main before merging:

$ git switch fix-shipping
$ git merge main
Auto-merging script.js
CONFLICT (content): Merge conflict in script.js

Both members edited the same file (script.js)

6

Resolving Merge Conflict

Member B resolves the conflict in script.js:

<<<<<<< HEAD
// Shipping calculation fix
function calculateShipping() {
// Member B's changes
}
=======
// Login validation fix
function validateLogin() {
// Member A's changes
}
>>>>>>> main

After manual editing, the file contains both fixes:

// Login validation fix
function validateLogin() {
// Member A's changes
}

// Shipping calculation fix
function calculateShipping() {
// Member B's changes
}
$ git add script.js
$ git commit -m "Merge main and resolve conflicts"
7

Final Merge to Main

Member B merges their fix to the main branch:

$ git switch main
$ git merge fix-shipping
Merge made by the 'ort' strategy.
script.js | 5 +++++
1 file changed, 5 insertions(+)

Both fixes are now in the main branch!

Key Takeaways

Benefits of Branching

  • Parallel development without conflicts
  • Isolated environments for each feature/fix
  • Ability to review changes before merging
  • Main branch always remains stable

When Conflicts Happen

  • Multiple people edit the same file
  • Git highlights conflicting sections
  • Manual resolution required
  • Communication with team members is key

Best Practice: Always pull the latest changes from the main branch before starting new work or creating a pull request. This minimizes merge conflicts and ensures your work is based on the most recent code.

Additional Git Information

Common Errors and Solutions

fatal: not a git repository (or any of the parent directories)

This error occurs when you're trying to run a Git command outside of a Git repository.

Solution: Run git init to initialize a new repository or navigate to an existing Git repository.

Please tell me who you are.

This happens when you haven't configured your user name and email.

Solution: Set your global username and email with git config --global user.name "Your Name" and git config --global user.email "your@email.com".

Useful Tips

  • Commit often with descriptive messages
  • Use git status frequently to check the state of your repository
  • Create branches for new features or experiments
  • Always pull the latest changes before starting work
  • Use .gitignore to exclude files that shouldn't be tracked
  • Use git diff to review changes before committing
  • Use git checkout -- <file> to discard changes in working directory
  • Prefer git rm and git mv over standard commands when working with Git
  • Use git switch -c <branch> to create and switch to a branch in one step
